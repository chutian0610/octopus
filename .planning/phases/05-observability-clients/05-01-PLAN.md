# DataFusion UDF/UDTF registration
DataFusion SessionContext supports UDF registration via `register_udf()`.
Scalar functions: create via `crate::functions::create_udf()`
Table functions (UDTF): create via `crate::functions::create_table_function()`
</context>

  <name>Task 1: Define UDF/UDTF registry trait and implementation</name>
  <files>octopus-common/src/udf.rs</files>
  <action>
Create `octopus-common/src/udf.rs` with:
- `UdfRegistry` trait with methods: `register_scalar(name, func)`, `register_table(name, func)`, `get_scalar(name)`, `get_table(name)`, `list_functions()`
- `UdfRegistryImpl` struct implementing the trait using `HashMap<String, ScalarUDF>` and `HashMap<String, TableFunction>`
- DataFusion type re-exports for: `ScalarUDF`, `TableFunction`, `ReturnTypeFunction`, `Signature`

<task type="auto">
  <name>Task 2: Integrate UDF registry with QueryService</name>
  <files>octopus-coordinator/src/query_service.rs</files>
  <action>
In `QueryService` (query_service.rs):
1. Add `udf_registry: Arc<RwLock<UdfRegistry>>` field to QueryService struct
2. Add `register_udf(name: &str, func: ScalarUDF)` method that delegates to registry
3. Add `register_table_function(name: &str, func: TableFunction)` method
4. Add `list_udfs()` method returning registered function names

After registration, when DataFusion parses SQL, registered functions are available
just like built-in functions. The QueryService context already parses SQL via
`SessionContext::sql()` which will find the registered UDFs.

Do NOT call `context.register_udf()` — instead maintain our own registry
per the D-04 decision for isolation. The registry stores functions and they
are referenced by name in queries.
  </action>
  <verify>cargo build -p octopus-coordinator 2>&1 | grep -v "^warning"</verify>
  <done>QueryService exposes UDF/UDTF registration methods; functions callable in SQL</done>
</task>
